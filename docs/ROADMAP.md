## 全体方針：ハイブリッド

- **集客**：ラボサイト（Astro / Notion / SEO・ブログ・占い師紹介）
- **予約〜決済〜育成**：公式LINEをハブに（離脱を最小化、人件費を最小化＝自動化）
- 「理想はNotionで」を尊重し、予約の空き枠管理などは可能な範囲でNotion＋GAS自前で持つ

```
[ラボ サイト(SEO・ブログ・占い師紹介)]  ← 発見・集客
        │ 友だち追加
        ▼
[ 公式LINE = ハブ ]
   ├ リッチメニュー「予約」→ 占い師別カレンダー(空き枠) → 前払い決済 → 確定通知
   ├ チャットボット(自動応答)でFAQ
   ├ タグ「占い師志望」→ ブランチへステップ配信
   └ 顧客カルテ(リピート施策)
```

## 未着手・今後やること

### ① ブログの個別化（優先度：次にやる）
**課題**：全占い師が1つのブログDBを共有 → 他人の記事も編集できてしまう。
**方針（決定：A案）**：占い師ごとに別ブログDBを作り、ビルドで複数DBを合算表示する。
- あなたの作業：占い師ごとにNotionブログDBを作成 → DB IDを共有
- 実装側：`src/lib/notion.ts` の `BLOG_DB`（単一）を「占い師→DB IDの配列/マップ」に変更し、`getPosts()` で全DBを合算。著者はDB＝占い師で確定できるので著者リレーションは簡略化可
- 代替（B案・不採用）：投稿フォーム経由で中央DBへ。占い師にDB編集権を渡さない

### ② 予約カレンダー（占い師ごとの空き枠）← コード実装済み・設定待ち
**方針（採用）**：Notion 占い師DBに営業時間を持たせ、GASが「営業枠 − 既存予約 − 過去時刻」を計算して予約フォームに返す（重複防止）。
- 実装済み：`gas/Availability.gs`（空き枠計算）、`gas/Code.gs`（doGetにslots API＋送信時の二重予約チェック）、`src/pages/reserve.astro`（占い師/日付の変更で空き枠を動的取得）
- **残作業（運用側）**：`docs/予約カレンダー設定.md` の手順で（1）占い師DBに 受付曜日/受付開始/受付終了/休業日 を追加、（2）GASを再デプロイ
- 未設定でも既定営業枠（毎日10:00〜21:30）で動くのでフォームは止まらない
- 将来：LINE内予約（LIFF）でも同じ空き枠APIを利用

### ③ 決済（前払い方式で no-show を削減）
**方針**：Stripe（Payment Links / Checkout）。サイトでもLINE webviewでも同じリンクで動く。Apple/Google Pay対応。手数料の目安 約3.6%（※要確認）。
- 予約確定時に前払い → 人手のノーショー対応を削減
- 将来 LINE Pay 追加も検討
- ノーコード希望なら STORES予約（決済つき）が代替
- **Stripeアカウントは後から切り替え可**（実装はAPIキー/リンクを差し替えるだけ。テスト→本番、法人化での付け替えも容易）

### ④ 公式LINE強化
- リッチメニュー（予約 / 占い師一覧 / ブログ / 占い師になりたい方へ）
- 自動応答（チャットボット）でFAQ。LINE公式の応答設定 or エルメ/Lステップ
- セグメント配信：「占い師志望」タグ → ブランチへステップ配信（ラボ→ブランチ導線の自動化）
- 顧客カルテ／リピート施策

### ⑤ 導線の追い込み（ラボ→ブランチ）

## 設定メモ（機微情報はコミットしない）

- `LIFF_ID`：LINE Developers で発行済み（consts.ts に設定）
- `LINE_URL`：公式アカウント友だち追加URL（consts.ts に設定）
- `NOTION_TOKEN`：`.env`（ローカル）/ GitHub Secrets（CI）。**コミット禁止**
- Stripe の秘密鍵を入れる場合も `.env` / Secrets のみ。ソースに直書きしない
